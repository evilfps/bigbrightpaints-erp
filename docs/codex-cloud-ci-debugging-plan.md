# CI Debugging + Accounting Verification Plan (Codex Cloud)

This plan documents the repeatable steps used to triage CI failures in Codex Cloud
and verify accounting integrity without introducing new features. It is aligned
with the current CI entrypoint and existing accounting services.

## Scope and current CI entrypoint (static review)
- CI runs `mvn -B -ntp verify` from `erp-domain` and publishes Surefire + JaCoCo
  artifacts for debugging (see `.github/workflows/ci.yml`). Local gate: `bash scripts/verify_local.sh` (must stay in sync
  with CI).

## CI failure investigation checklist (Codex Cloud)
1. Locate the failing tests
   - Open the `erp-domain` job logs to find the first failing test and capture the
     failure stack trace.
   - Download `erp-domain/target/surefire-reports/**` artifacts to identify all
     failing tests and any setup/teardown errors from XML + TXT reports.
2. Map failures to suite
   - Categorize failing tests by suite directory (e.g., `invariants`,
     `regression`, `modules`, `smoke`) to determine intent and scope.
3. Configuration cross-check
   - Validate test-time settings (disabled schedulers, JWT secrets, etc.) match
     Codex Cloud runtime configuration in `application-test.yml`.
4. Classify failure type
   - Deterministic logic error (assertion mismatch, validation message change).
   - Nondeterministic failure (time/ordering, concurrency, race conditions).
   - Infra/config (missing DB, env, secrets, migrations).
   - Data/setup (seed data or fixtures diverging).
5. Fast local reproduction
   - If not environment-specific, reproduce locally with
     `mvn -Dtest=<FailingTest> test`.

## Long-running tasks (async execution guidance)
For long tasks in Codex Cloud, run them asynchronously and track progress so the
agent can continue working without blocking.

1. Start the task in the background and capture the PID:
   - `nohup mvn -B -ntp verify > /tmp/ci-verify.log 2>&1 & echo $! > /tmp/ci-verify.pid`
2. Poll the log for progress:
   - `tail -n 100 /tmp/ci-verify.log`
3. Check completion status:
   - `ps -p $(cat /tmp/ci-verify.pid)`
   - When the process exits, inspect the log and exit code if captured by wrapper.
4. Record outcomes:
   - Note failures with exact test names and stack traces.
   - Attach Surefire/JaCoCo artifacts if available.

## Accounting verification checklist (industry-standard)

### 1) Double-entry integrity and idempotency
- Journal entries enforce balancing (base currency) and include FX rounding
  tolerance with correction to the largest debit/credit line when within
  tolerance.
- Duplicate reference protection returns existing entries and prevents double
  postings.

Verification steps
- Validate every journal entry balances to zero after rounding adjustment; include
  FX/non-FX permutations.
- Confirm duplicate reference behavior is idempotent and consistent for manual
  vs. auto-generated references.

### 2) Period close and temporal consistency
- Journal entry posting enforces open periods when period lock is enabled;
  reversals require open or authorized override.
- Accounting period close creates closing entries and locks periods; reopen
  auto-reverses closing entries.
- Temporal balances reconstruct historical balances using accounting events.

Verification steps
- Validate period lock enforcement with and without admin override.
- Ensure temporal trial balance matches ledger balances for historical dates.

### 3) Chart of accounts and hierarchy
- Balance sheet and income statement hierarchies are built from account trees
  with total rollups.

Verification steps
- Verify parent/child account type constraints and consolidated totals match
  expectations.

### 4) AR/AP aging and statements
- Dealer aging buckets use due-date-based bucket logic; statements compute
  opening balance + running balance.

Verification steps
- Validate bucket boundaries (0/30/60/90/90+ days) and due-date logic.
- Ensure statements reconcile to ledger totals (opening + period movements =
  closing).

### 5) Tax and reference integrity
- GST return sums journal lines for configured tax accounts.
- Reference numbers are generated by sequence with audit trail.

Verification steps
- Ensure tax accounts are configured and tax totals align with invoice ledger
  postings.
- Confirm reference sequences are unique and auditable across modules.

### 6) Reconciliation checks
- AR/AP reconciliations compare GL balances to dealer/supplier sub-ledgers with
  tolerance.

Verification steps
- Ensure GL totals match sub-ledger totals within tolerance; investigate
  discrepancies by partner.

## Cross-module linkage verification
Use the existing linkage matrix as the authoritative expected linkage for O2C,
P2P, Production/Inventory, Payroll, and Reversals.

Key automated linkages to verify
- Inventory valuation/movement events auto-post journal entries and enforce
  idempotent references.
- Journal entries update dealer/supplier ledgers when posted with partner
  context.

## Cross-module missing logic / debugging watchlist
1. Account hierarchy totals
   - Account hierarchy service computes consolidated balances via recursive
     summation; ensure report/UI usage depends on the computed totals, not raw
     node balances.
2. Aging vs. statement alignment
   - Aging reports bucket by due date; statements track running balance by entry
     date. Validate discrepancies are expected (e.g., prepayments or credits) and
     documented in tests.
3. Period enforcement bypass conditions
   - Idempotent journal entry handling skips period validation for existing
     references; ensure upstream services never reuse references for different
     data.
4. Event-driven inventory journaling
   - Inventory events skip posting when source/destination accounts are missing;
     ensure callers supply accounts or document manual posting requirements.
5. Reversal + period lock interaction
   - Reversal checks reject locked periods and require override for closed
     periods; confirm tests cover reversal vs. close/reopen flows.

## Execution checklist for Codex Cloud runs
- Capture CI logs + Surefire artifacts.
- Classify failures per section above; capture reproduction steps.
- For accounting-related failures, verify against the checklist and link to the
  relevant class/logic.
- Update the report with concrete failing tests and root causes once CI output is
  available.
