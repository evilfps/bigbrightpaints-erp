# Database, Persistence, and Migration Posture

Last reviewed: 2026-03-30

This packet documents the database layer of the orchestrator-erp backend: persistence technology, schema areas, entity/repository conventions, migration strategy, profile-based activation, legacy-track constraints, and data-import entry surfaces. It fulfills VAL-PLAT-005.

A maintainer or frontend collaborator should be able to understand the active migration posture, the conventions governing schema and repository code, and the surviving import surfaces without reading Flyway SQL files or Spring configuration by hand.

## 1. Persistence Technology

- **Database:** PostgreSQL (driver: `org.postgresql.Driver`)
- **ORM:** Spring Data JPA / Hibernate
- **DDL mode:** `spring.jpa.hibernate.ddl-auto = none` — Hibernate never creates or alters tables at runtime. All schema changes go through Flyway migrations.
- **Connection pooling:** HikariCP (default pool: 20 max, 5 min idle; production overrides via `SPRING_DATASOURCE_POOL_MAX` / `SPRING_DATASOURCE_POOL_MIN`)
- **Default schema:** `public` (explicit in `application-prod.yml`)

### 1.1 Optimistic locking

All entities that extend `VersionedEntity` carry a `version` column with a JPA `@Version` annotation. This provides optimistic concurrency control without requiring pessimistic database locks. The `VersionedEntity` base class lives at:

- `core/domain/VersionedEntity.java`

The version column is `bigint NOT NULL DEFAULT 0` and is enforced at the JPA level, not through a database trigger.

### 1.2 ID generation

Entities use `GenerationType.IDENTITY` — the database generates IDs via PostgreSQL sequences (`bigint` columns with `NEXT VALUE FOR` sequence ownership). Each table has a corresponding `_<table>_id_seq` sequence.

## 2. Entity and Repository Conventions

### 2.1 Entity conventions

| Convention | Detail |
| --- | --- |
| Base class | Most mutable entities extend `VersionedEntity` for optimistic locking |
| ID type | `Long` (bigint), generated by database identity sequences |
| Public IDs | Some entities carry a `publicId` (`UUID`, auto-generated via `gen_random_uuid()`) for external-facing references |
| Tenant scoping | Almost all domain tables include a `company_id bigint NOT NULL` foreign key to `companies`. JPA queries filter by `companyId` resolved from the security context. This is the primary multi-tenant isolation mechanism. |
| Timestamps | Most entities include `created_at timestamptz NOT NULL DEFAULT now()`. Some also carry `updated_at`. |
| Soft delete | Not used as a global convention. Deletion is physical or governed by module-specific lifecycle rules (e.g., `enabled` flag on users, lifecycle state on companies). |
| Column naming | `snake_case` in DDL, mapped to `camelCase` fields via Hibernate implicit naming. |
| Precision | Monetary amounts use `numeric(19,4)`. Quantities typically use `numeric(19,4)` or `integer`. |

### 2.2 Repository conventions

All repositories are Spring Data JPA interfaces:

```java
public interface SomeRepository extends JpaRepository<SomeEntity, Long> { ... }
```

- Some repositories also extend `JpaSpecificationExecutor` for dynamic query building (e.g., `JournalEntryRepository`).
- Queries are defined via Spring Data method naming conventions, `@Query` annotations, or `Specification` objects.
- Repositories are package-scoped within each module's `domain/` package.
- Cross-module repository access is discouraged; modules that need data from another module should go through a facade or service boundary. The architecture allowlist governs exceptions.

## 3. Major Schema Areas

The v2 migration track organizes schema creation by domain phase. The baseline migrations are:

| Migration | Domain Area | Key Tables |
| --- | --- | --- |
| `V1__core_auth_rbac.sql` | Auth, RBAC, audit, companies | `app_users`, `companies`, `roles`, `permissions`, `user_roles`, `audit_logs`, `audit_log_metadata`, `blacklisted_tokens`, `mfa_recovery_codes`, `password_reset_tokens` |
| `V2__accounting_core.sql` | Accounting | `accounts`, `journal_entries`, `journal_lines`, `accounting_periods`, `accounting_events`, `accounting_period_snapshots`, `partner_settlement_allocations`, `dealer_ledger_entries`, `supplier_ledger_entries` |
| `V3__sales_invoice.sql` | Sales, invoice, dealer | `sales_orders`, `sales_order_items`, `invoices`, `invoice_lines`, `dealers`, `credit_limit_override_requests`, `dealer_ledger_entries`, `packaging_slips` |
| `V4__inventory_production.sql` | Inventory, production, factory | `catalog_imports`, `production_products`, `raw_materials`, `raw_material_batches`, `finished_good_batches`, `inventory_movements`, `factory_tasks`, `production_logs`, `packing_records`, `size_variants` |
| `V5__purchasing_hr.sql` | Purchasing, HR, payroll | `suppliers`, `purchase_orders`, `purchase_order_lines`, `goods_receipts`, `purchase_invoices`, `employees`, `payroll_runs`, `payroll_run_lines` |
| `V6__orchestrator.sql` | Orchestrator, outbox, scheduler | `outbox_events`, `orchestrator_commands`, `audit_records`, `scheduled_job_definitions` |
| `V7__enterprise_audit_ml_events.sql` | Enterprise audit trail, ML events | `audit_action_events`, `ml_interaction_events` |

Subsequent migrations (V8–V172) add indexes, idempotency columns, lifecycle fields, module gating, company-scoped constraints, catalog variants, dispatch logistics, closed-period exceptions, HR/Payroll module pause, opening stock batch-key alignment, superadmin control-plane schema, auth-v2 scoped accounts, finished-goods bulk-flag removal, and supplier outstanding balance hard cuts.

### 3.1 Cross-module FK posture

The v2 track creates foreign keys within the same domain area in baseline migrations. Some cross-module FKs are deferred to later migrations. Key cross-module relationships:

- `journal_entries.company_id` → `companies.id`
- `sales_orders.company_id` → `companies.id`
- `inventory_movements.company_id` → `companies.id`
- `outbox_events.company_id` → `companies.id`
- Module-local FKs (e.g., `journal_lines.journal_entry_id` → `journal_entries.id`) are created within the same baseline migration.

Cross-module FKs that span baseline boundaries (e.g., sales → accounting, purchasing → inventory) are added in subsequent migrations after both sides of the relationship exist.

## 4. Migration Strategy: Flyway V2 Hard-Cut

### 4.1 Active migration track

**Flyway v2 (`db/migration_v2/`) is the only active migration track.** All new schema changes must be added as v2 migrations. The v1 track (`db/migration/`) is legacy and must not receive new migrations.

Key properties:

| Property | Value |
| --- | --- |
| Flyway enabled | `true` in all profiles |
| V2 migration location | `classpath:db/migration_v2` |
| V2 history table | `flyway_schema_history_v2` |
| V2 profile activation | `flyway-v2` (included in the `prod` profile group by default) |
| Default profile group | `prod` includes `flyway-v2` |

### 4.2 Profile activation

The `flyway-v2` profile is activated automatically in production through the Spring profile group mechanism:

```yaml
# application.yml
spring:
  profiles:
    group:
      prod:
        - flyway-v2
```

This means `SPRING_PROFILES_ACTIVE=prod` automatically activates `flyway-v2`. Explicit `SPRING_PROFILES_ACTIVE=prod,flyway-v2` also works (and is used in Docker Compose deployments).

The `flyway-v2` profile configures:

```yaml
# application-flyway-v2.yml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration_v2
    table: flyway_schema_history_v2
  datasource:
    url: ${SPRING_DATASOURCE_V2_URL:${SPRING_DATASOURCE_URL}}
    username: ${SPRING_DATASOURCE_V2_USERNAME:${SPRING_DATASOURCE_USERNAME}}
    password: ${SPRING_DATASOURCE_V2_PASSWORD:${SPRING_DATASOURCE_PASSWORD}}
```

**Environment variable override:** V2 migration datasource can use separate credentials (`SPRING_DATASOURCE_V2_*`) that fall back to the standard datasource variables when not set.

### 4.3 Profile-specific Flyway behavior

| Profile | Flyway enabled | Migration location | History table | Notes |
| --- | --- | --- | --- | --- |
| `prod` (+ implicit `flyway-v2`) | `true` | `db/migration_v2` | `flyway_schema_history_v2` | Production active track |
| `test` | `true` | `db/migration_v2` (default) | `flyway_schema_history_v2` (default) | Uses v2 by default; env overrides available |
| `seed` + `flyway-v2` | `true` | `db/migration_v2` | `flyway_schema_history_v2` | Seed/demo data profile |
| `openapi` | `false` | — | — | Uses H2 in-memory; no Flyway |
| `benchmark` | `true` | `db/migration` (legacy) | default | **Caveat:** benchmark profile still points at the legacy track. This is intentional for benchmark isolation but should not be used as a template for new work. |

### 4.4 MIGRATION_SET environment variable

Shell scripts and CI commands use `MIGRATION_SET=v2` to signal the active migration set for build and test workflows. This variable is consumed by:

- `scripts/verify_local.sh` — defaults to `v2`, validates against the matching migration set.
- `scripts/schema_drift_scan.sh` — defaults to `v1`, must be explicitly set to `v2` for v2 drift scans.
- Maven commands — passed as environment context for migration-aware test execution.

All CI and local-development Maven commands should set `MIGRATION_SET=v2`.

### 4.5 Forward-only posture

Many v2 migrations are **intentionally forward-only**. They alter data in ways that cannot be reversed with a simple SQL undo script. Examples:

- `V168/V169` — rewrite auth accounts to be company-scoped; cannot reverse the normalized email/scope split.
- `V167` — introduce superadmin control-plane schema with lifecycle enum rewrite and column rename.
- `V166` — rewrite opening-stock batch keys and drop legacy replay fingerprints.
- `V171` — drop the legacy `finished_good_batches.is_bulk` flag.

Recovery from a failed migration requires **database snapshot or point-in-time recovery (PITR)** rather than Flyway undo scripts. This posture is documented in [ADR-005](../adrs/ADR-005-flyway-v2-hard-cut-migration-posture.md) and enforced through:

- Runbook requirements: every migration must update [docs/runbooks/migrations.md](../runbooks/migrations.md) and [docs/runbooks/rollback.md](../runbooks/rollback.md).
- Enterprise policy checks: high-risk migration paths under `db/migration_v2/` require R2 checkpoint evidence in [docs/approvals/R2-CHECKPOINT.md](../approvals/R2-CHECKPOINT.md).

## 5. Legacy Migration Track

### 5.1 Status and constraints

The `db/migration/` directory contains **158 legacy migration files** (V1 through V165, with some gaps). These files:

- Are **preserved for reference** — they contain historical context about the original schema design.
- **Must not receive new migrations.** The architecture decision in [ADR-005](../adrs/ADR-005-flyway-v2-hard-cut-migration-posture.md) is unambiguous: all new work targets the v2 track.
- Are **not executed** by the active application configuration. The `flyway-v2` profile points at `db/migration_v2`, not `db/migration`.
- May share **some overlapping migration numbers** with v2 migrations in later ranges (e.g., V157/V158 exist in both tracks but carry different content). This is a known historical artifact from when certain migrations were applied to both tracks before the hard cut.

### 5.2 Dual-track cleanup history

Some migrations that were originally applied to the v1 track were subsequently recreated or forward-ported to the v2 track. This means:

- The v2 track is **self-contained**: a fresh database can be bootstrapped using only v2 migrations.
- The v1 track is **not needed** to bring up a new environment.
- The `benchmark` profile still points at the legacy track for isolation purposes, but this should not be treated as a signal that the legacy track is active.

## 6. Data-Import Entry Surfaces

Several modules expose bulk data-import endpoints that matter for backend understanding. These endpoints handle file uploads (CSV or XML), perform idempotency-protected imports, and persist row-level results.

### 6.1 Accounting imports

| Endpoint | Method | Module | Purpose |
| --- | --- | --- | --- |
| `POST /api/v1/accounting/opening-balances` | Multipart file upload | accounting | Import opening balance journal entries from CSV. Admin-only (`ROLE_ADMIN`). Idempotency key + company-scoped unique constraint on `opening_balance_imports`. |
| `POST /api/v1/migration/tally-import` | Multipart file upload | accounting | Import Tally XML data into the accounting system. Admin-only (`ROLE_ADMIN`). Idempotency key + company-scoped unique constraint on `tally_imports`. |

Entity conventions for accounting imports:

- `OpeningBalanceImport` extends `VersionedEntity`, carries `companyId`, `idempotencyKey`, `fileHash`, `status`, and row-level results.
- `TallyImport` extends `VersionedEntity`, carries `companyId`, `idempotencyKey`, `fileHash`, `status`, and row-level results.
- Both have `@UniqueConstraint(columnNames = {"company_id", "idempotency_key"})` at the JPA level.

### 6.2 Catalog import

| Endpoint | Method | Module | Purpose |
| --- | --- | --- | --- |
| `POST /api/v1/catalog/import` | Multipart file upload | production | Import brands, items, raw materials from CSV. Admin-only. Idempotency key + company-scoped unique constraint on `catalog_imports`. |

Entity: `CatalogImport` extends `VersionedEntity`, carries `companyId`, `idempotencyKey`, `idempotencyHash`, `fileHash`, `fileName`, import statistics, and `errorsJson`.

### 6.3 Opening stock import

| Endpoint | Method | Module | Purpose |
| --- | --- | --- | --- |
| `POST /api/v1/inventory/opening-stock` | Multipart file upload | inventory | Import opening stock batches from CSV. Admin-only. |
| `GET /api/v1/inventory/opening-stock` | — | inventory | Retrieve opening stock import history. |

Entity: The opening-stock import path uses `opening_stock_imports` with `company_id`, `idempotency_key`, `replay_protection_key` (legacy), `opening_stock_batch_key` (canonical), and `results_json` for row-level results. Import idempotency is handled by `OpeningStockImportService` with batch-key + idempotency-key dual-key replay protection.

### 6.4 Import caveats

- All import endpoints require `ROLE_ADMIN` — they are not available to dealer or portal users.
- Import services perform their own idempotency checks before processing; the shared `IdempotencyReservationService` is not always used directly (some imports use module-local import-table unique constraints instead).
- Import results are persisted on the import record (row-level `results_json` or equivalent), so historical imports can be audited.
- The Tally import uses the `/api/v1/migration/` host prefix rather than `/api/v1/accounting/`. This is a historical convention that has not been migrated to the accounting host.

## 7. Scope and Default-Value Caveats

### 7.1 company_id is non-optional on domain tables

Almost every domain table includes `company_id bigint NOT NULL`. Multi-tenant isolation depends on this column being present and non-null. Queries that do not filter by `company_id` would return cross-tenant data, which the application guards against via:

- `CompanyContextFilter` resolving the current tenant from the JWT
- Repository queries explicitly filtering by `companyId`
- JPA `where` clauses on tenant-scoped entities

A few infrastructure tables (e.g., `app_users` in the auth scope, `shedlock`) do not carry `company_id` because they are platform-scoped rather than tenant-scoped.

### 7.2 ddl-auto = none

Hibernate is configured with `ddl-auto = none` in all profiles. This means:

- No schema is ever generated or altered by Hibernate at runtime.
- All DDL changes must go through Flyway migrations.
- Missing columns or tables will cause runtime errors rather than silent schema creation.

This is a deliberate choice: the migration track is the single source of truth for schema state.

### 7.3 Benchmark profile legacy track

The `benchmark` profile (`application-benchmark.yml`) points Flyway at `classpath:db/migration` (the legacy track) with `clean-disabled: false`. This is intentional for benchmark isolation but means the benchmark schema may diverge from the production v2 schema over time. Do not use the benchmark profile as a reference for schema conventions.

### 7.4 Numeric precision

Monetary amounts use `numeric(19,4)` throughout the v2 schema. This provides sufficient precision for INR-denominated financial calculations. The precision choice is applied consistently across accounting, sales, purchasing, and payroll tables.

### 7.5 Timestamps

Most tables use `timestamp with time zone` (`timestamptz`) for `created_at` and similar columns, defaulting to `now()`. This ensures timezone-aware timestamps in PostgreSQL.

## 8. Migration Safety and Operational Notes

### 8.1 Coordinated app-and-schema cuts

Many v2 migrations are designed as **coordinated app-and-schema cuts**: the migration and the corresponding application code change must be deployed together. Redeploying older application code against a database that has already run a hard-cut migration is explicitly unsupported.

The rollback runbook ([docs/runbooks/rollback.md](../runbooks/rollback.md)) documents per-migration rollback strategies, which typically require:

1. Keep the current (post-migration) backend live.
2. Restore the database from a pre-migration snapshot/PITR.
3. Only then redeploy the pre-migration backend.

### 8.2 CI safety gates

- `ci/check-enterprise-policy.sh` — detects high-risk migration paths and requires R2 checkpoint evidence.
- `scripts/guard_legacy_migration_freeze.sh` — prevents new migrations from being added to the legacy track.
- `scripts/schema_drift_scan.sh` — detects drift between expected and actual schema state.
- `scripts/verify_local.sh` — validates the full local build against the active migration set.

### 8.3 Flyway history table separation

The v2 track uses `flyway_schema_history_v2` as its history table, separate from the default `flyway_schema_history` used by the legacy track. This allows the two tracks to coexist in the same database without interfering with each other's version tracking, though the legacy track is no longer executed in production.
